# llama.cpp로 로컬 LLM 벤치마킹

> RTX 4060 + llama.cpp 환경에서 Llama 3 / Mistral / Qwen2 모델의 토큰 처리 속도와 양자화 품질을 직접 비교.

---

## 배경

회사에서 TensorRT로 비전 모델을 FP16/INT8로 변환·벤치마킹하던 경험을 LLM에도 적용해 보고 싶었다.
"양자화로 얼마나 빨라지는가, 품질은 얼마나 잃는가" — 같은 질문이었다.

---

## 환경

```
GPU  : RTX 4060 (8GB VRAM)
CPU  : Intel Core i7-13700
RAM  : 32GB DDR5
OS   : Windows 11 + WSL2 (Ubuntu 22.04)
도구 : llama.cpp (CUDA 빌드) · ollama · lm-evaluation-harness
```

---

## 테스트 대상

| 모델 | 파라미터 | 양자화 |
|---|---|---|
| Llama 3.1 8B | 8B | Q4_K_M / Q8_0 / F16 |
| Mistral 7B v0.3 | 7B | Q4_K_M / Q8_0 |
| Qwen2.5 7B | 7B | Q4_K_M / Q8_0 |

---

## 토큰 처리 속도 (tokens/sec, prompt 512 tokens)

| 모델 | Q4_K_M | Q8_0 | F16 |
|---|---|---|---|
| Llama 3.1 8B | **68.4** | 41.2 | 22.1 |
| Mistral 7B | **71.3** | 43.7 | 23.8 |
| Qwen2.5 7B | **66.9** | 40.1 | — (VRAM 초과) |

> F16 기준 Q4_K_M이 약 3배 빠름. TensorRT FP16→INT8 가속비(1.5~2x)보다 훨씬 크다.
> LLM은 연산보다 메모리 대역폭이 병목이기 때문.

---

## 품질 비교 (ARC-Easy, HellaSwag 기준)

| 모델 | Q8_0 | Q4_K_M | 차이 |
|---|---|---|---|
| Llama 3.1 8B | 81.2% | 80.7% | **-0.5%p** |
| Mistral 7B | 79.4% | 78.9% | **-0.5%p** |
| Qwen2.5 7B | 80.1% | 79.3% | **-0.8%p** |

손실이 1%p 미만이면 실사용에서 체감하기 어렵다.
비전 모델의 INT8 양자화가 mAP 1~3%p 하락하는 것보다 훨씬 관대하다.

---

## 정성 평가: 한국어 응답 품질

코딩 질문·요약·번역 각 10개를 직접 채점(1~5점).

| | Llama 3.1 8B Q4 | Mistral 7B Q4 | Qwen2.5 7B Q4 |
|---|---|---|---|
| 한국어 자연스러움 | 3.2 | 2.8 | **4.3** |
| 코딩 (Python) | **4.1** | 3.9 | 4.0 |
| 요약·번역 | 3.5 | 3.2 | **4.4** |

한국어 작업엔 Qwen2.5가 압도적. Llama / Mistral은 영어 편향이 강하게 느껴진다.

---

## 결론

- **속도 우선 + 한국어**: Qwen2.5 7B Q4_K_M
- **영어 코딩**: Llama 3.1 8B Q4_K_M
- Q4_K_M이 사실상 "로컬 LLM 기본값"으로 적절하다. Q8_0은 VRAM 여유가 충분할 때만.

---

## 비전 모델 최적화와 비교해 보니

TensorRT INT8은 캘리브레이션 데이터셋 구성이 까다롭고 레이어별로 정밀도를 잡아야 한다.
llama.cpp 양자화는 명령어 한 줄이면 끝나고 품질 저하도 훨씬 작다.
LLM 쪽이 양자화 내성이 훨씬 높다는 게 체감으로 와 닿았다.
